# Бэкенд и синхронизация (фундамент) — дизайн

**Дата:** 2026-06-25
**Статус:** утверждён, готов к плану
**Зависит от:** существующего PWA-трекера (localStorage), задеплоенного на GitHub Pages.

## Цель

Перенести хранение данных трекера с локального `localStorage` в **базу на личном сервере пользователя**, чтобы данные были надёжными и доступными с разных устройств. Это фундамент, поверх которого позже лягут интеграции FatSecret (крон) и шаги/калории (Apple Health → Shortcut).

## Ограничения

- **Без поддомена и без изменений DNS.** На сервере уже работает другой проект; добавлять лишнее нельзя.
- **Минимальный след на сервере:** один Node-процесс + один файл SQLite + одна `location`-секция в уже существующем reverse-proxy.
- **Код PWA публичный** (GitHub Pages, публичный репозиторий) → в репозитории не должно быть секретов. Адрес API и токен вводятся в приложении и хранятся в браузере.
- **HTTPS обязателен:** PWA отдаётся по HTTPS, браузер заблокирует запросы на http (mixed content). Переиспользуем HTTPS существующего домена.

## Архитектура

```
[PWA, GitHub Pages]  ──HTTPS + Bearer-токен──►  https://домен/trackerapi/*
   (Safari, айфон)                                      │  (reverse-proxy: одна location)
   localStorage = офлайн-кэш                            ▼
                                              127.0.0.1:8787  [Node API]
                                                              │
                                                              ▼
                                                        tracker.sqlite  [SQLite]
```

- Reverse-proxy (nginx или Caddy), который уже терминирует HTTPS для домена, проксирует **только** префикс `/trackerapi/` на локальный порт Node. Остальные маршруты домена не затрагиваются.
- Node-сервис слушает только `127.0.0.1:8787` (наружу не торчит, ходит через прокси).

## Компоненты

### 1. Сервер — `server/`
- `server/index.js` — HTTP-API (встроенный модуль `http`, без тяжёлых фреймворков).
- `server/db.js` — обёртка над SQLite (модуль `node:sqlite`, встроен в Node ≥ 22; если версия ниже — `better-sqlite3`).
- `server/auth.js` — проверка Bearer-токена (сравнение в постоянное время).
- `server/config.js` — чтение настроек из переменных окружения (`PORT`, `TRACKER_TOKEN`, `DB_PATH`, `ALLOW_ORIGIN`).
- `server/package.json` — `"type": "module"`, скрипты `start` и `test`.
- `server/tracker.service` — пример systemd-юнита.
- `server/README.md` — инструкция по деплою (nginx и Caddy сниппеты, выпуск токена, запуск).

### 2. Клиентский слой синхронизации — `src/sync.js`
- Инкапсулирует общение с API: `pull()`, `push(state)`, проверку доступности, хранение `apiUrl`/`token` (в localStorage, не в коде).
- Встраивается в существующий `src/storage.js`: локальное хранилище остаётся источником на лету, `sync.js` зеркалит на сервер.

## API

Базовый префикс: `/trackerapi`. Все ответы — JSON. Все, кроме healthcheck, требуют заголовок `Authorization: Bearer <token>`.

| Метод | Путь | Назначение | Тело / ответ |
|---|---|---|---|
| GET | `/trackerapi/health` | healthcheck (без токена) | `{ ok: true }` |
| GET | `/trackerapi/state` | получить состояние | `{ state: <json|null>, updatedAt: <int ms> }` |
| PUT | `/trackerapi/state` | сохранить состояние | вход `{ state, updatedAt }` → ответ `{ updatedAt }` |

- `PUT` сохраняет переданное состояние и проставляет серверный `updatedAt = max(полученный, now)`.
- Ошибки: `401` без/с неверным токеном, `400` на кривой JSON, `413` на слишком большое тело (лимит, напр. 2 МБ).

> **Важно про префикс:** прокси срезает `/trackerapi` (nginx `proxy_pass .../`, Caddy `handle_path`), поэтому **Node реализует маршруты без префикса** — `/health`, `/state`. Полный путь `/trackerapi/...` существует только снаружи, для клиента. Не дублировать префикс в коде Node.

## Модель данных (SQLite)

```sql
CREATE TABLE IF NOT EXISTS state (
  id         INTEGER PRIMARY KEY CHECK (id = 1), -- одна строка: один пользователь
  json       TEXT NOT NULL,
  updated_at INTEGER NOT NULL                    -- эпоха, мс
);
```

Одна строка (`id = 1`) хранит весь JSON-объект состояния трекера — ту же структуру, что сейчас в localStorage. Нормализацию в таблицы не делаем: данных мало, запросов по полям не нужно (YAGNI). Точки расширения (FatSecret, health) добавят свои таблицы позже.

## Семантика синхронизации

- В состоянии заводим поле `updatedAt` (мс), обновляется при каждом локальном изменении.
- **При запуске приложения:** `pull()` с сервера. Если `server.updatedAt > local.updatedAt` → берём серверное, пишем в localStorage и перерисовываем. Иначе оставляем локальное и `push()` его на сервер.
- **При каждом изменении:** `push()` на сервер с задержкой (debounce ~800 мс), чтобы не слать на каждый тап.
- **Правило разрешения конфликтов:** последний по `updatedAt` побеждает (last-write-wins). Для одного пользователя этого достаточно; настоящий merge не делаем.
- **Офлайн:** если сеть/сервер недоступны — продолжаем работать на localStorage, помечаем «не синхронизировано», повторяем при следующем изменении или запуске. Данные не теряются.
- **Индикатор:** маленький статус синка в интерфейсе (синхронизировано / нет сети / ошибка), без модалок.

## Авторизация

- Один длинный случайный токен (генерится при деплое, напр. `openssl rand -hex 32`), хранится на сервере в переменной окружения `TRACKER_TOKEN`.
- В приложении (Настройки) — поля **«Адрес API»** и **«Ключ»**; сохраняются в localStorage. В каждый запрос подставляется `Authorization: Bearer <token>`.
- Секрет **не попадает в репозиторий** (вводится пользователем в приложении).
- Сравнение токена — в постоянное время, чтобы не утекал по таймингам.

## Выставление наружу и деплой (без поддомена)

- Node слушает `127.0.0.1:8787` (наружу закрыт).
- В существующий конфиг reverse-proxy добавляется одна секция для префикса:

  **nginx** (внутри существующего `server { ... }` домена):
  ```nginx
  location /trackerapi/ {
      proxy_pass http://127.0.0.1:8787/;
      proxy_set_header Host $host;
      proxy_set_header X-Forwarded-Proto $scheme;
  }
  ```

  **Caddy** (внутри существующего блока домена):
  ```
  handle_path /trackerapi/* {
      reverse_proxy 127.0.0.1:8787
  }
  ```
- Префикс `/trackerapi/` выбран уникальным, чтобы не пересечься с маршрутами текущего проекта (при необходимости сменим на ещё более редкий).
- Node-сервис запускается как systemd-юнит (автозапуск, рестарт при падении).

## CORS

- Бэкенд отвечает `Access-Control-Allow-Origin: https://sedrcoin.github.io` (origin Pages), `Access-Control-Allow-Headers: Authorization, Content-Type`, `Access-Control-Allow-Methods: GET, PUT, OPTIONS`, и корректно отрабатывает preflight `OPTIONS`.
- Origin берётся из переменной `ALLOW_ORIGIN` (на будущее, если домен Pages сменится).

## Безопасность

- Только HTTPS (через прокси).
- CORS ограничен origin'ом Pages.
- Лимит размера тела запроса (напр. 2 МБ).
- Простой rate-limit по IP (защита от перебора токена), напр. N запросов/мин.
- SQLite-файл в каталоге сервиса, права только владельцу.

## Тестирование

- **Сервер:** интеграционные тесты через `node --test`, поднимающие сервер на случайном порту и бьющие реальными HTTP-запросами: healthcheck без токена; 401 без/с неверным токеном; PUT→GET роундтрип; last-write-wins по `updatedAt`; 400 на битый JSON; 413 на превышение лимита; CORS-заголовки и preflight.
- **БД:** тесты обёртки `db.js` на временном файле (get/set, инициализация схемы).
- **Клиент (`sync.js`):** чистые части (выбор «серверное/локальное» по `updatedAt`, формирование запроса) — юнит-тесты с мок-fetch.

## Что нужно от пользователя (этап деплоя)

- Что на сервере уже терминирует HTTPS для домена: **nginx** или **Caddy** (или другое).
- Возможность добавить одну `location`/`handle_path` секцию и поднять Node-сервис (systemd) — самостоятельно по инструкции или предоставив доступ.
- Версия Node на сервере (для выбора `node:sqlite` vs `better-sqlite3`).

## Вне рамок этого этапа (позже)

- Интеграция FatSecret (крон, OAuth) — отдельная спека.
- Шаги/калории из Apple Health через Shortcut (`POST /trackerapi/health`) — отдельная спека.
- Мультипользовательность, реальная многоустройственная синхронизация с merge, realtime.

## Критерии готовности

1. Node-сервис поднимается локально и проходит все тесты (`npm test` в `server/`).
2. С айфона (PWA на Pages) после ввода адреса API и ключа в Настройках: изменения сохраняются на сервер и подтягиваются при следующем запуске.
3. Очистка данных Safari / другое устройство → после ввода ключа данные восстанавливаются с сервера.
4. Офлайн: приложение работает на localStorage, по возвращении сети синхронизируется, ничего не теряется.
5. Существующий проект на домене не затронут (добавлена только одна проксируемая секция на префиксе).
